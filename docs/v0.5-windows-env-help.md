# javahome 0.5 Windows environment and context-aware help

Version 0.5 adds a Windows-specific persistent environment mode and context-aware help output.

## Context-aware help

`javahome help` now detects the current platform and shell and shows the relevant platform section first.

Examples:

- Linux shows Linux/Bash/Zsh/Fish examples and Linux discovery notes.
- macOS shows macOS/Zsh examples and `/usr/libexec/java_home` discovery notes.
- Windows shows PowerShell and Windows user/machine environment examples.

This avoids showing Windows instructions in a Linux terminal or Linux-specific profile instructions in a Windows terminal.

## Windows persistent environment

For the current PowerShell only, use the existing shell activation flow:

```powershell
javahome use 17 --shell powershell | Invoke-Expression
```

For future PowerShell sessions through the PowerShell profile:

```powershell
javahome use 17 --global --shell powershell
```

For real Windows user environment variables:

```powershell
javahome windows-env user 17
```

For machine-wide Windows environment variables:

```powershell
javahome windows-env machine 17
```

Machine scope usually requires an elevated Administrator terminal.

Preview first:

```powershell
javahome windows-env user 17 --dry-run
javahome windows-env machine 17 --dry-run
```

## Reboot behavior

A reboot is normally not required.

However, Windows processes inherit environment variables when they start. Already-running terminals, IDEs, daemons, and services will not automatically change their current process environment.

After `windows-env`:

- open a new terminal
- restart IDEs such as IntelliJ or Eclipse
- restart build daemons such as Gradle daemon if needed
- restart Windows services if they depend on Java
- reboot only when restarting affected processes manually is inconvenient

`windows-env` also sends a Windows environment-change notification after writing the variables, but already-running programs still decide themselves whether they reload environment changes.

## What gets changed

`windows-env` sets:

- `JAVA_HOME`
- scoped `Path`

It prepends `%JAVA_HOME%\bin` equivalent to the selected JDK path and removes obvious old Java/JDK/JRE `bin` entries from the selected scope's Path.

Supported scopes:

```text
user
machine
```

## Examples

Use Java 17 for the current PowerShell only:

```powershell
javahome use 17 --shell powershell | Invoke-Expression
```

Set Java 17 persistently for the current Windows user:

```powershell
javahome windows-env user 17
```

Set Java 21 machine-wide:

```powershell
Start-Process powershell -Verb RunAs
javahome windows-env machine 21
```

Use a vendor filter:

```powershell
javahome windows-env user 17 --vendor temurin
```
